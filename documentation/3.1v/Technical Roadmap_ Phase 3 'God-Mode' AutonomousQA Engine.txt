Technical Roadmap: Phase 3 'God-Mode' AutonomousQA Engine
1. The 'God-Mode' Paradigm Shift: Deterministic Engineering
The transition from "Normal" AI—characterized by high-latency, probabilistic LLM dependencies—to Phase 3 "God-Mode" represents an architectural manifesto for 100% deterministic engineering. We are moving away from asking models if a UI "looks right" and toward mathematically provable truth. By replacing expensive, hallucination-prone API calls with localized algorithmic scoring, we achieve microsecond execution speed and absolute reliability.Table 1: Normal AI (Probabilistic) vs. Phase 3 God-Mode (Deterministic)| Dimension | Normal AI Approach (Probabilistic) | Phase 3 God-Mode (Deterministic) || ------ | ------ | ------ || Speed | Network I/O Bound (4000ms+ API latency) | CPU Bound (Microsecond/15ms execution) || Cost | High ($ per API token per run) | $0.00 (Offline & Local execution) || Reliability | Probabilistic (Hallucinations possible) | 100% Deterministic (Mathematical Truth) || Self-Healing | LLM-driven DOM repair | Fuzzy DOM Scoring (Levenshtein + Pythagorean) || Vision | Gemini-driven visual analysis | SSIM, Pixel Math, and Gaussian Blurring |
Deterministic Logic Overrides
* Visual Regression Math:  Traditional Mean Squared Error (MSE) approaches fail due to pixel-level noise from font anti-aliasing. Our "Extraordinary" engine utilizes  Blurred Image Subtraction . By applying a Gaussian blur to both baseline and current screenshots, we eliminate microscopic hex variations while preserving macro-structural integrity. We then use Pillow.ImageChops.difference() to calculate statistical variance.
* Algorithmic Healing:  We utilize a  Mathematical Scoring Matrix  to identify elements when selectors break. This is not a "guess"; it is a heuristic summation:
* Tag Match:  +20 points.
* Text Similarity (Levenshtein Distance):  Up to +35 points for character-level proximity.
* Attribute Intersection:  Up to +25 points for CSS class/ID overlap.
* Spatial Proximity (Pythagorean Spatial Decay):  Up to +20 points based on 2D coordinate drift.
2. Implementation A+C: GitHub Metadata Engine & Advanced UI
The GitHub Engine enables "Shift Left" testing, allowing autonomous validation of code states before they reach a production URL.
OAuth Workflow (Stage 0)
* Redirect to GitHub:  The Gateway redirects users to the GitHub OAuth portal requesting the repo and user:email scopes.
* Token Exchange:  Upon user approval, the temporary code is exchanged for a githubAccessToken, persisted in the Database Layer.
* Repository Synchronization:  The engine uses the token to fetch a list of all public and private repositories accessible to the user.
Metadata Extraction Logic
The RepoManager (located in ai-core/utils/repo_server.py) captures developer-centric context immediately following a shallow clone (--depth 1).
* Command:  git log -1 --pretty=format:"%H|%s|%an|%ae"
* Captured Data:  Commit SHA (Full), Commit Message, Author Name, and Author Email.
* Branch Identification:  Captured via git rev-parse --abbrev-ref HEAD.
Frontend UI Specifications
The Dashboard and History pages utilize  Next.js 16.1  for microsecond responsiveness.
* GitHub Chip:  A premium UI element for repo-mode runs. It features a GitHub icon, the branch name, and a  Short SHA Pill (7 characters) .
* Contextual Linking:  The report header provides a direct "View on GitHub" link mapping to the specific commit SHA.
3. RepoGraph: Tree-sitter AST to Neo4j Mapping
RepoGraph provides a structured representation of the repository, enabling fine-grained context for autonomous agents without ingesting the entire codebase.
Construction Logic
1. Code Line Parsing:  The engine traverses the repository, using  Tree-sitter  to generate an Abstract Syntax Tree (AST). We selectively retain only those lines involving function calls and dependency relations, discarding extraneous boilerplate and individual redundant variables.
2. Project-Dependent Filtering:  We exclude "Global" (Python built-ins like len, list) and "Local" (third-party imports identified via import statement parsing) relations to minimize noise.
3. Graph Organization:  The repo is mapped as  $G = \{V, E\}$ .
Graph Schema Definition
Element,Type,Description
Node (  $V$  ),def,"Line where a function, class, or variable is defined. Attributes: file, line, category."
Node (  $V$  ),ref,"Line where an entity is invoked. Attributes: file, line, method."
Edge (  $E$  ),Einvoke,Directed edge from a definition to a reference.
Edge (  $E$  ),Econtain,"Directed edge connecting a module (e.g., class) to its internal components."
Neo4j Retrieval
The engine retrieves  k-hop ego-graphs  centered on specific search terms. This captures immediate functional relationships, allowing the agent to understand the "blast radius" of a potential defect without loading irrelevant files into the context window.
4. KISS Sorcar: Git Worktree Isolation & Dev-Server Orchestration
KISS Sorcar ensures that every test run occurs in a pristine, isolated environment.
Isolation & Setup
The system executes a git clone --depth 1 into a unique directory per run. This shallow clone provides the necessary files without the overhead of full repository history.
Project Detection Mapping
The engine identifies the stack based on root-level manifest files:| Root File | Project Type | Dev Start Sequence || ------ | ------ | ------ || package.json | Node.js | npm install -> npm run dev || pyproject.toml | Python | pip install . -> python -m app || Cargo.toml | Rust | cargo run || go.mod | Go | go run . || index.html | Static | python3 -m http.server |
Dev-Server Logic
The engine monitors stdout for URL patterns (e.g., http://localhost:5173).
1. Timeout:  If no URL is detected within  15 seconds , the engine fallbacks to the provided PORT environment variable.
2. Warm-up:  Once the URL is found, a  2-second warm-up period  is enforced before the crawler agent is dispatched.
5. Architectural Blueprints & File System Mapping
4-Layer Tech Stack
1. Client Layer:  Next.js 16.1 (App Router), shadcn/ui, and Framer Motion for a premium, high-performance interface.
2. API Gateway:  Express.js handling NextAuth.js, Prisma ORM, and Socket.io WebSocket relays.

7. Feature-Specific Deep Dives & Validation Protocols
Feature 1: Self-Healing (Fuzzy Algorithmic Engine)
* Validation:  Modify a button's id and text (e.g., "Submit" to "Confirm Order") in the dev server.
* Protocol:  Verify the engine re-identifies the target based on its spatial coordinate and tag similarity. The click must resolve in  < 15ms .
Feature 3: Visual Regression (Blurred Image Subtraction)
* Validation:  Introduce a 1px shift in a navigation menu.
* Protocol:  The engine must apply Gaussian blurring to baseline/current states. Fail the test only if the "Drift Percentage" calculated by ImageChops exceeds the  0.5% noise threshold .
Feature 5: Performance Chaos Agent
* Validation:  Inject network throttling (3G) using the Chaos Agent.
* Protocol:  Measure LCP and FID against the unthrottled baseline. The engine must generate a "Performance Regression" alert.
